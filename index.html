Intelligent CSS Components

Evolutionary

eBay's cohesive multi-screen experience starts with a shared and unified design system. As this system evolves, so too does Skin, meaning you only need to keep your Skin package updated to ensure the latest look and feel.

Adaptive

Screens 7" and below are considered to be small screens; they are our default, baseline experience. Screens over 7" are large screens. Skin provides support for these screen sizes in an adaptive manner.

Accessible

Accessibility is at the heart of Skin. A widget must not only look like a certain type of widget, it must be described like that widget too. Skin leverages ARIA roles, states and properties to apply our styles wherever possible, thus enforcing correct, accessible semantics.

Readable

Skin follows the BEM principles of Block, Element and Modifier to ensure our class name and class structure is human readable and understandable. The princicples of BEM also act as a front-line defense against classitis.

Decoupled

Skin is exclusively a CSS toolkit, meaning that JavaScript is not included as part of this project. Remaining decoupled from the JavaScript layer means that our HTML and CSS should not need to be reimplemented every time the JS frontend framework changes.

Open

Skin operates under an open source model. If you wish to contribute, raise an issue, or ask a question, head over to GitHub for more information.

Adapters

Skin provides two mechanisms in order to support adaptive use cases for large screens & legacy browsers.

Skin-Large

Skin is built for small screens first (7" and below). Styles for large screens are applied in an adaptive manner when the skin-large class is found on the document body.

Skin itself does not do any detection of screen size, therefore it is the responsibility of the application logic to detect whether the skin-large class is needed.

Skin-Legacy

Skin takes advantage of LassoJS conditional dependencies to provide cross-browser support for legacy browsers. We currently define legacy browsers as those that do not support Flexbox or SVG.

To plugin this conditional dependency, pass the flag skin-legacy to Lasso and apply the skin-legacy class to the body of your page.

Skin itself does not do any detection of browser or browser capabilities, therefore it is the responsibility of the application logic to detect whether the skin-legacy class is needed.

IE9

Sometimes we will need specific CSS rules for older versions of IE. We try our best to keep these to a minimum, but we ask that you use the following code at the top of your HTML file to ensure that the correct CSS hooks are in place for IE9.

Alert

All users* need clear feedback to understand what’s blocking them from progress; either when they’ve been successful or simply when there’s more information available to shop and sell successfully. An alert focuses on reducing cognitive load and making the next steps in the journey clear.

An alert may have one of three possible levels: priority, confirmation, or information.

Dependencies: skin-ebay/iconfont.

*If an alert is rendered on the client and/or updated on the client, the entire alert must be wrapped with an additional div or span. This div or span must contain attributes role="status" & aria-live="polite". This 'live-region' wrapper is necessary in order for assistive technology to detect & announce a subtree change.

Use the page-alert--priority, page-alert--confirmation, or page-alert--information modifier classes depending on the level of alert required.

Flyout Alert

Flyout alerts are available on large, 7+ screens only. Like inline alerts, they call out a specific section or element of the page.

Content order of flyout alerts is important! To avoid reading-order and tabbing-order accessibility issues, the flyout element should be placed adjacent in the DOM to the page section or element it points to.

Flyout alerts are absolutely positioned by default. The flyout will be positioned relatively to the nearest ancestor with position:relative.

Use the flyout-alert--priority, flyout-alert--confirmation, or flyout-alert--information modifier classes depending on the level of alert required.

The flyout pointer can be positioned using any one of the following modifier classes: flyout-alert--top-left, flyout-alert--top-center, flyout-alert--top-right, flyout-alert--bottom-right, flyout-alert--bottom-center, flyout-alert--bottom-left, flyout-alert--left-middle or flyout-alert--right-middle.

Atom

An atom is a basic unit of style, representing a single CSS property (or shorthand property).

Some of these atoms are applied directly by our Experience Service to represent business logic (e.g. item-hotness). Others are more commonly applied via client-side logic (e.g. hide and show).

Button

A button may be used to submit a form, reset a form or trigger any custom action via a JavaScript click event (e.g. to fetch new results, open a dialog or expand a menu).

A button may be created using the input tag or button tag. In some special cases, an anchor tag may be used to create a link that appears visually like a button (see the Faux Button section below for more details).

Button Modifiers

The btn base class is required for every button. Optional modifier classes exist for hierarchy, size, width and action behaviour. One modifier of each type may be applied.

Button Hierarchy

The default button hierarchy is regular and requires no class modifier.

For primary or secondary buttons, use the .btn--primary or .btn--secondary hierarchy modifiers.

Button Size

The default button size is medium and requires no class modifier.

For small or large buttons, use the .btn--small or .btn--large size modifiers.

Fluid Button

A fluid button fills the entire width of it's parent element. Please use the .btn--fluid width modifier.

Faux Button

A faux button is an anchor tag styled to appear visually as a button. Faux buttons should be used sparingly, and only for prominent call-to-action links, e.g. 'Start Shopping', 'Take me There', etc.

The btn--faux modifier class is required. The href attribute must always contain a valid URL. A value of "javascript" (or any such variant) is not a valid URL.

Shop Now

Hijaxed Faux Button

A faux button can have it's link behaviour progressively enhanced into button behaviour. For example, the link might be enhanced to open the destination URL inside of an overlay instead of leaving the page. Because AJAX is typically used, this technique is often referred to as 'hijax' (i.e. hi-jacking a link).

The btn--hijax modifier class and button role are both required.

The href attribute must always contain a valid fallback URL. A value of "javascript" (or any such variant) is not a valid URL.

View Item

Faux Link

A faux link is a button tag styled to appear visually as a link. Warning: We do not encourage this use case, because it can cause confusion for our users. Users typically expect something that looks like a link to open a new URL, rather than running a JavaScript action.

Disabled Button

The disabled attribute is required to fully disable any button.

Disabled Faux Button

Remove the href attribute to fully disable an anchor tag.

Buy it Now

Partially Disabled Button

Partially disabled buttons are visually and aurally identifiable as disabled, but remain keyboard focusable. Partially disabled buttons can vastly simplify keyboard focus management logic in situations where the button may frequently toggle back and forth between enabled and disabled (e.g. pagination or carousel bookend buttons).

The aria-disabled state is required for partially disabled buttons.

Icon Buttons

Use any of the available icon fonts to display an icon on a button.

An icon on a button with no text is deemed a critical icon. An aria-label property is required.

An icon on a button with text is deemed a non-critical icon, and no aria-label is necessary.

Card

A card forms the content layer in the eBay Design System topography of background layer, content layer & content. This system requires that the background layer (typically the HTML body tag) have zero margin and padding. The card content layer is responsible for creating all necessary margin & padding.

There are three types of card: Regular Card, Primary Card & Full Page Card.

All cards use a nested card__cell element to denote the inner content of the card block. Any content placed outside of this cell will be full-bleed (i.e. edge to edge). See Hero Card section below for more information.

NOTE: in all card examples below we use a fixed-height div to simulate Skin's default grey background layer.

Regular Card

Use the card class to create a card with vertical and horizontal margins.

This 'floating' effect helps differentiate data sets and is used to identify secondary, tertiary & extraneous content.

Dialog

A dialog is a child window spawned by the main web page or application.

The main body of the dialog forms the background layer, and has zero margin and padding by default (similar to the body tag). Content may be placed directly onto the background layer, inside of a dialog__cell element. Alternatively a content layer can be created using the card module and/or grid module.

Skin dialogs are always modal, meaning that mouse, keyboard & screen reader interactions with the dialog's parent window must be prevented while the dialog is open.

A dialog must always be labelled using the aria-labelledby or aria-label properties, and must contain at least one focusable, interactive element.

Dependencies: skin-ebay/iconfont.

On small screens the default dialog is full-page.

For a close button with text (e.g. "Cancel", "Done") use class dialog__close and apply inner text.

For a close button with default icon ('X'), use modifier class dialog__close--icon and apply aria-label.

Grid BETA

The eBay grid is a flexbox first layout system, used to give to structure and form to designs. It will enable building consistent layouts across multiple pages.

For all Screens

By using our grid system pages can be made to adapt to any screen size. It uses media queries in order to change the layout. However, builds will be included for both adaptive and responsive use cases.

Consistent Design

The large and medium grid layouts are based on fractions of sixteen; the small grid is fractions of eight. Each grid cell has a percentage width with fixed "gutters". Having fixed spacing allows for consistent rhythm in our vertical and horizontal spacing.

Customizable

Currently the maximum width of a layout is 1280 pixels, but this system can easily be customized to fit all needs.

Backwards Compatible

In order to support older browsers a set of fallbacks classes were added.

Getting Started

There are two ways to get started using the grid system. The first, is to included the bundled grid css file into a link tag in the head of your document as show below.

Lasso Flags

If you are using lasso, you can included the grid css by adding a flag to your lasso configuration. For this to work you must include the grid.browser.json file to your browser.json as a dependency. If no flags are specified the grid-core file will be loaded into your page.

  • grid-core

    Adds the core grid system to your page. This includes just includes the lg, sm, and default grid sizes.

  • grid-large

    In order to accommodate adaptive designs the large and extra-large classes were broken up into their own file. This allows for applications to only pull in the large and extra-large classes

  • grid-small

    In order to accommodate adaptive designs the small classes were broken up into their own file. This allows for applications to only pull in the small classes

  • grid-full

    Adds the entire grid system to your page. This includes the lg, sm and the default grid sizes. As well as the necessary classes to utilize nested grids.

Basic Grid

  • Create an element and apply the .grid class.
  • The only direct children of the .grid element should be .grid__group elements.
  • Generally, the only direct children of the .grid__group element should be .grid__cell elements. The only current exception is when you use the .grid__group--full modifier.
  • Generally, the .grid__cell element should have one block element child.

In order to simplify the grid system classes, the default classes to layout a page are .grid__cell--*of*, replacing the * with a number up to 16.

Listbox

A listbox (also known as a select) allows the user to select one item from a list of options.

A listbox is typically used to collect form data and therefore must always be used in conjunction with a form and submit button. If you are not submitting form data or do not wish to reload the entire page, then a Menu maybe a better choice.

Listbox Modifiers

The base class for all listboxes is listbox. Modifiers exist for label, size & border. One modifier of each type may be applied.

Default Listbox

The default listbox is a full-width, block level element with label on top.

Side-Label Listbox

To align the label alongside the listbox, apply the listbox--side-label modifier.

Unselected Listbox

To force a user to make selection, the first option can be used as a prompt and set to disabled.

Disabled Listbox

A disabled field is conveyed using the disabled attribute. Disabled field values are not passed to the server.

Note that the listbox--disabled modifier is also needed.

Unlabelled Listbox

For listboxes without a visible label, use aria-label to provide an accessible label to assistive technology.

Note that the listbox--no-label modifier class must also be provided.

Small Listbox

For a smaller listbox, apply listbox--small modifier.

Borderless Listbox

For a borderless listbox, apply the listbox--borderless modifier.

Borderless listboxes have a side label by default.

If dropping the visual label, use listbox--no-label.

Tab

A tabbed interface allows the layered stacking of content panels, with only one panel being visible at any time. A tab is contained within a tablist - each tab controlling the visibility of it's respective panel. By decluttering the user-interface in this way, we say that a tab follows the principals of progressive disclosure.

When adding a tabbed interface to a page, you have two choices: selecting a tab can either update the panel contents dynamically without a full page reload (these are our default tabs), or they can behave statically, reloading the entire page (we call these our 'Faux' Tabs). Be sure to use the correct version for your needs!

Tab Modifiers

The base class for all tab modules is tabs. A single modifier exists for tab action behaviour.

Default Tabs

When a tab is selected, the aria-selected, aria-hidden and tabindex states must be updated in order for the CSS to reflect the change. Only one tab can be selected and in the tab order at any moment in time. Likewise, only one tabpanel can be visible at any time, and it must correspond to the currently selected tab.

Textbox

A textbox can be single-line or multi-line, using the input or textarea element, respectively.

Textbox Modifiers

The base class for all textboxes is textbox. Modifiers exist for label, size & icon. One modifier of each type may be applied.

Default Textbox

The default textbox is a full-width, block level element with label on top.

Multi-line Textbox

A multi-line textbox allows line breaks.

Side-Label Textbox

To align the label alongside the textbox, use the textbox--side-label modifier.

NOTE: Label length is restricted to 25 characters.

Wrapped Side-Label Textbox

To wrap side label text to two lines on large screens, use the textbox--wrapped-side-label modifier.

NOTE: Label length is restricted to 52 characters and truncation with ellipses requires JavaScript.

Textbox Placeholders

Placeholder text is intended as a short hint towards the type of value that should be entered. Both single-line and multi-line textboxes can have placeholder text.

WARNING: Because placeholder text is transient (i.e. it is cleared on user input), it does not qualify as an accessible label. Always use a real label (onscreen or offscreen) in tandem with placeholder text.

Hidden Label Textbox

The aria-label property can be used to provide a non-visual label that remains accessible to assistive technology. To prevent possible cognitive difficulties for our users, a hidden label is only acceptable if there is some other visual clue (e.g. an icon or placeholder text) as to the purpose of the field.

Remember too that placeholders are not an accessible alternative to visual labels, the only possible exception to this rule is for very short forms (e.g. a login form, as in the example below), and even then a hidden label is still always required.

Textbox Icons

Skin provides a set of textbox icons that can be used simply by applying the relevant modifier class to the input. For example textbox--username or textbox--search. These icons are currently only compatible with single-line textboxes. If you do not find the textbox icon that you need, let us know and we will consider adding it to our set.

Small Textbox

For a smaller textbox (including any label and icon), apply the .textbox--small modifier class.

Disabled Textbox

A disabled field is conveyed using the disabled attribute. Disabled field values are not passed to the server.

Readonly Textbox

A readonly field is conveyed using the readonly attribute. Readonly field values do get passed to the server.

Invalid Textbox

An error is conveyed using the aria-invalid state. Additional context for the error is conveyed using the aria-describedby property which must point to the unique id of the error message container. This error message container can be designated as a live region if client-side input validation is implemented and enabled.
